# Product Requirements Document: ensembl-orm

## 1. Project Overview

### 1.1 Purpose

Create a Python 3.13 read-only ORM for the Ensembl MySQL database (`homo_sapiens_core`), providing a typed, validated interface for the HGNC release pipeline. The library maps 7 Ensembl tables and resolves the current release database name dynamically.

### 1.2 Context

The HGNC release pipeline (Phase 2) imports cross-reference data from 27 external sources. Four of those sources and the coordinate builder query the Ensembl MySQL database for gene coordinates, cross-references, and cytoband data. This ORM provides the data access layer for those consumers.

### 1.3 Consumers

| Consumer | Sources / Sub-sources | Tables Used |
|---|---|---|
| `hgnc-xref-loader` | cytoband, ensembl2hgnc, ensembl2hgnc_complete, ensembl_gene | `gene`, `seq_region`, `xref`, `object_xref`, `external_db`, `seq_region_attrib`, `karyotype` |
| `hgnc-coord-builder` | EnsemblGeneCoords | `gene`, `seq_region` |

### 1.4 Success Criteria

- All 7 Ensembl tables mapped with correct column types and names
- Dynamic database name resolution from current Ensembl release
- Read-only session management with connection pooling
- Comprehensive test coverage using mocks (no real DB in CI)
- Installable as a git-source dependency via uv

## 2. Technical Architecture

### 2.1 Core Technology Stack

| Component | Choice | Rationale |
|---|---|---|
| ORM framework | SQLModel 0.0.33+ | Combines SQLAlchemy 2.0 + Pydantic; matches genew4-orm pattern |
| Database driver | mysqlclient | MySQL driver; required by AGENTS.md |
| Configuration | Pydantic BaseSettings | Matches genew4-orm; env-var driven |
| Python version | 3.13+ | Matches parent project |
| Package manager | uv | Team standard |

### 2.2 Key Differences from genew4-orm

| Aspect | genew4-orm | ensembl-orm |
|---|---|---|
| Database | PostgreSQL (genew4) | MySQL (homo_sapiens_core) |
| Driver | psycopg | mysqlclient |
| Access | Read-write with audit | Read-only (anonymous) |
| Database name | Fixed (`genew4`) | Dynamic (`homo_sapiens_core_{VERSION}_{ASSEMBLY}`) |
| Schema management | Alembic migrations | None (external database) |
| Session types | Read-only + read-write | Read-only only |

## 3. Configuration

### 3.1 DatabaseSettings

A Pydantic `BaseSettings` class that loads connection parameters from environment variables with the `ENSEMBLDB_` prefix.

| Setting | Env Var | Default | Description |
|---|---|---|---|
| `host` | `ENSEMBLDB_HOST` | `ensembldb.ensembl.org` | Ensembl MySQL hostname |
| `port` | `ENSEMBLDB_PORT` | `5306` | Ensembl MySQL port |
| `user` | `ENSEMBLDB_USER` | `anonymous` | Database username |
| `password` | `ENSEMBLDB_PASSWORD` | (empty string) | Database password |
| `database` | `ENSEMBLDB_DATABASE` | (empty) | Explicit database name (bypasses discovery when set) |
| `pool_size` | `ENSEMBLDB_POOL_SIZE` | `3` | Connection pool size |
| `pool_recycle` | `ENSEMBLDB_POOL_RECYCLE` | `3600` | Recycle connections after N seconds |

### 3.2 Connection URL Format

```text
mysql+mysqldb://{user}:{password}@{host}:{port}/{database}
```

### 3.3 Environment Variable Example

```bash
ENSEMBLDB_HOST=ensembldb.ensembl.org
ENSEMBLDB_PORT=5306
ENSEMBLDB_USER=anonymous
ENSEMBLDB_PASSWORD=
```

When `ENSEMBLDB_DATABASE` is not set, the discovery module resolves it automatically.

## 4. Database Discovery

### 4.1 Purpose

The Ensembl database name includes the release version and assembly (e.g., `homo_sapiens_core_112_38`). The current release version changes periodically, so the ORM must resolve the database name dynamically.

### 4.2 Resolution Process

1. Fetch the current Ensembl release version number from `http://ftp.ensembl.org/pub/VERSION` (plain text, e.g., `112`)
2. Connect to the Ensembl MySQL server (without specifying a database)
3. Execute `SHOW DATABASES LIKE 'homo_sapiens_core_{version}%'` to find the exact database name
4. Return the first matching database name (e.g., `homo_sapiens_core_112_38`)

### 4.3 Caching

The resolved database name is cached in memory for the lifetime of the process. Subsequent calls to `discover_database_name()` return the cached value without re-querying.

### 4.4 Override

If `ENSEMBLDB_DATABASE` is set in the environment, discovery is skipped entirely. This allows consumers to pin to a specific Ensembl release for testing or reproducibility.

### 4.5 Error Handling

| Scenario | Behaviour |
|---|---|
| Version fetch fails (network error) | Raise `EnsemblDiscoveryError` with the underlying exception |
| Version fetch returns non-integer | Raise `EnsemblDiscoveryError` with descriptive message |
| No matching database found | Raise `EnsemblDiscoveryError` listing attempted pattern |
| Multiple matching databases | Log warning, use first match (sorted alphabetically) |

## 5. Session Management

### 5.1 Read-Only Only

All sessions are read-only. The Ensembl database is an external public resource with anonymous access -- there is no write use case.

### 5.2 API Surface

```python
import ensembl_orm

engine = ensembl_orm.initialize_engine()

with ensembl_orm.get_session() as session:
    genes = session.exec(select(Gene).limit(10)).all()

ensembl_orm.close_all_sessions()
```

### 5.3 Public Functions

| Function | Signature | Description |
|---|---|---|
| `initialize_engine` | `(settings: DatabaseSettings \| None = None) -> Engine` | Create the global engine, running discovery if needed |
| `get_engine` | `() -> Engine` | Return the global engine (raises if not initialized) |
| `get_session` | `() -> Generator[Session]` | Context manager yielding a read-only session |
| `close_all_sessions` | `() -> None` | Dispose engine and clear factories |

### 5.4 Engine Lifecycle

1. `initialize_engine()` is called once at application startup
2. If `settings.database` is not set, run `discover_database_name()` to resolve it
3. Create a SQLAlchemy engine with `mysql+mysqldb` dialect and connection pooling
4. All sessions use the same engine
5. `close_all_sessions()` disposes the engine on shutdown

## 6. Database Models

### 6.1 Mapping Strategy

All models use SQLModel with `table=True`, mapping to existing Ensembl tables. Column names use `sa_column` to map Python attribute names to actual MySQL column names where they differ.

Key principles:
- One model class per file, named in snake_case matching the class (e.g., `SeqRegion` -> `seq_region.py`)
- All fields typed with `| None` for nullable columns
- Primary keys use `Field(default=None, primary_key=True)`
- No DDL generation -- `metadata.create_all()` must never be called

### 6.2 Table: `gene`

Used by xref-loader (EnsemblGene, Ensembl2Hgnc, Ensembl2HgncComplete, EnsemblGeneCoords) and coord-builder.

| Column | Type | Nullable | Description |
|---|---|---|---|
| `gene_id` | `Integer` | PK | Primary key |
| `biotype` | `String(255)` | No | Gene biotype (e.g., `protein_coding`) |
| `analysis_id` | `Integer` | No | FK to analysis table |
| `seq_region_id` | `Integer` | No | FK to seq_region |
| `seq_region_start` | `Integer` | No | Genomic start position |
| `seq_region_end` | `Integer` | No | Genomic end position |
| `seq_region_strand` | `Integer` | No | Strand (1 or -1) |
| `display_xref_id` | `Integer` | Yes | FK to xref for display label |
| `source` | `String(255)` | No | Source of gene annotation |
| `description` | `Text` | Yes | Gene description |
| `is_current` | `Boolean` | No | Whether this is the current gene version |
| `canonical_transcript_id` | `Integer` | Yes | FK to transcript |
| `canonical_translation_id` | `Integer` | Yes | FK to translation |
| `stable_id` | `String(128)` | Yes | Ensembl stable ID (e.g., `ENSG00000139618`) |
| `version` | `Integer` | Yes | Version number |
| `created_date` | `DateTime` | Yes | Record creation date |
| `modified_date` | `DateTime` | Yes | Record modification date |

Relationships:
- `seq_region` -> `SeqRegion` (many-to-one)

Define Python enums for the following Ensembl column types:

| Enum | Values | Used By |

## 7. Error Handling

### 7.1 Custom Exceptions

| Exception | Raised When |
|---|---|
| `EnsemblDiscoveryError` | Version fetch fails, no matching database found, or invalid version |
| `EnsemblSessionError` | Engine not initialized before session creation |

### 7.2 Logging

Use Python `logging` module with the `ensembl_orm` logger name:

| Level | Events |
|---|---|
| `debug` | Connection URL (password masked), SQL queries |
| `info` | Database name resolved, engine initialized |
| `warning` | Multiple databases matched during discovery |
| `error` | Version fetch failure, connection failure |

## 8. Testing Strategy

### 8.1 Approach

All tests use mocks -- no real database connections. This follows AGENTS.md rules: "Tests must be isolated -- never interact with a real database or external service."

### 8.2 Test Categories

**Configuration tests** (`test_database_settings.py`):
- Default values load correctly
- Environment variables override defaults
- Connection URL is built correctly (with and without password)
- Validation rejects invalid port numbers

**Session tests** (`test_session.py`):
- `initialize_engine()` creates engine with correct dialect
- `initialize_engine()` calls discovery when database not set
- `initialize_engine()` skips discovery when database is explicitly set
- `get_session()` yields a usable session
- `get_session()` raises `EnsemblSessionError` if engine not initialized
- `close_all_sessions()` disposes engine and clears factories
- `initialize_engine()` returns existing engine on repeated calls

**Discovery tests** (`test_discovery.py`):
- Fetches version from correct URL
- Parses version number from plain text response
- Executes `SHOW DATABASES` with correct LIKE pattern
- Returns first matching database name
- Caches resolved database name
- Raises `EnsemblDiscoveryError` on network failure
- Raises `EnsemblDiscoveryError` when no matching database
- Skips discovery when database is explicitly configured
- Logs warning on multiple database matches

**Model tests** (`models/test_*.py`):
- Each model maps to the correct table name
- All columns are defined with correct types
- Nullable columns accept `None`
- Composite primary keys are configured correctly (`seq_region_attrib`)
- Relationships are defined correctly where applicable
- Enum fields accept only valid values

### 8.3 Test Fixtures

`conftest.py` provides:

- `mock_settings`: A `DatabaseSettings` instance with test values
- `mock_engine`: A mocked SQLAlchemy Engine
- `mock_session`: A mocked SQLAlchemy Session
- `mock_version_response`: A mocked HTTP response with version number
- `mock_show_databases_result`: A mocked `SHOW DATABASES` result set

### 8.4 Test Execution

```bash
uv run pytest tests/ -v
```

## 10. Implementation Phases

### Phase 1: Project scaffolding and configuration

- Create project directory structure
- Write `pyproject.toml` with all dependencies
- Write `.gitignore`, `.env.example`, `.markdownlint.json`
- Implement `DatabaseSettings` in `config/database_settings.py`
- Implement `__init__.py` with public API exports
- Write tests for `DatabaseSettings`

### Phase 2: Database discovery

- Implement `discover_database_name()` in `discovery.py`
- Fetch release version via `urllib.request` from `http://ftp.ensembl.org/pub/VERSION`
- Connect to MySQL and resolve database name via `SHOW DATABASES`
- Implement caching of resolved name
- Implement skip-when-explicitly-set logic
- Define `EnsemblDiscoveryError` in `exceptions.py`
- Write tests for discovery (all mocked)

### Phase 3: Session management

- Implement `session.py` with engine factory and read-only session context manager
- Implement `initialize_engine()`, `get_engine()`, `get_session()`, `close_all_sessions()`
- Integrate with `DatabaseSettings` and `discovery` module
- Define `EnsemblSessionError` in `exceptions.py`
- Write tests for session management (all mocked)

### Phase 4: Enums

- Define `EnsemblObjectType`, `ExternalDbStatus`, `InfoType`, `ExternalDbType` using `StrEnum`
- Place in `src/ensembl_orm/enums.py`
- Write tests for enum values

### Phase 5: Models -- gene and seq_region

- Implement `Gene` model in `models/gene.py`
- Implement `SeqRegion` model in `models/seq_region.py`
- Define relationship: `Gene.seq_region` -> `SeqRegion`
- Write model tests

### Phase 6: Models -- xref, object_xref, external_db

- Implement `ExternalDb` model in `models/external_db.py`
- Implement `Xref` model in `models/xref.py`
- Implement `ObjectXref` model in `models/object_xref.py`
- Define relationships: `Xref.external_db` -> `ExternalDb`, `ObjectXref.xref` -> `Xref`

### Phase 7: Models -- seq_region_attrib and karyotype

- Implement `SeqRegionAttrib` model with composite PK in `models/seq_region_attrib.py`

### Phase 8: Integration verification and polish

- Verify all models import correctly through `models/__init__.py`
